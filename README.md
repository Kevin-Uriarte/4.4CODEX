# Actividad de GitHub Classroom: Propuesta de Práctica Temática Pequeña

## Descripción general
En esta actividad vas a **diseñar y documentar** una propuesta de práctica temática pequeña orientada a arquitectura de computadoras o programación de sistemas. El proyecto puede implementarse en **ARM64 Assembly, C, Python o Bash**, pero la prioridad es que primero entregues una propuesta clara, viable y bien estructurada.

## Objetivo de aprendizaje
Al finalizar esta actividad, serás capaz de:

- Definir un problema técnico pequeño y acotado.
- Justificar la elección de lenguaje según el caso.
- Diseñar la estructura de un repositorio limpio para GitHub Classroom.
- Documentar caso de uso, alcance, riesgos y pruebas mínimas.
- Construir (opcionalmente) un prototipo mínimo alineado con tu documentación.

## Lenguajes permitidos
- ARM64 Assembly
- C
- Python
- Bash

## Reglas para mantener el proyecto pequeño
- Enfócate en **una sola funcionalidad principal**.
- Limita el tiempo estimado de implementación a pocas horas.
- Evita interfaces gráficas; usa terminal.
- No uses frameworks, servicios externos, nube o bases de datos.
- Si eliges ARM64 Assembly, mantén el alcance en programas muy pequeños.
- Prioriza claridad y calidad de documentación sobre cantidad de código.

## Entregables esperados
1. `docs/propuesta.md` completo.
2. `docs/caso_de_uso.md` completo.
3. `docs/estructura_repositorio.md` revisado para tu proyecto.
4. `docs/plan_de_pruebas.md` con casos definidos.
5. `tests/test_plan.md` con checklist final.
6. Prototipo mínimo opcional en `src/main.<ext>`.

## Instrucciones para el estudiante
1. Lee todo el repositorio antes de programar.
2. Elige tema y lenguaje principal.
3. Llena primero los archivos de `docs/`.
4. Define alcance mínimo y límites claros.
5. Si te alcanza el tiempo, implementa un prototipo pequeño.
6. Ejecuta o adapta `scripts/run.sh` para validar estructura y ejecución básica.
7. Entrega con commits claros y ordenados.

## Criterios generales de evaluación
- Claridad de la propuesta y del problema a resolver.
- Viabilidad técnica del alcance.
- Justificación del lenguaje elegido.
- Coherencia entre caso de uso, pruebas y estructura.
- Organización del repositorio y calidad de documentación.
- Evidencia de ejecución mínima (si aplica).

## Ejemplos de posibles temas
- **Mini Toolkit en ARM64**
- **Asistente de Estudio en Terminal**
- **Reporteador de Información del Sistema**
- **Organizador de Archivos**
- **Juego de Aprendizaje en Línea de Comandos**

## Nota importante
Primero se documenta y se justifica la idea. Después, de forma opcional, se implementa un prototipo pequeño. La calidad de la planeación vale más que escribir mucho código.
